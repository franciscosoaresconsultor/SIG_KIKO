O SIG KIKO foi concebido como um Sistema de Apoio à Decisão (SAD) que mitiga a perda de oportunidades de negócio e maximiza a eficiência no fecho de acordos.

RN1

Proibição de acordos fantasma

A assinatura de um Acordo carece de pelo menos uma visita validada e com presença confirmada. Implementado em VBA via evento Form_BeforeUpdate.

Qualificação rigorosa

O preço por m² nunca é digitado nem armazenado: é calculado em runtime via SQL/formulários computados. Respeita a 3FN e elimina dependência transitiva.

VBA · O Porteiro Digital

Código original do projeto

Quando as Validation Rules nativas se esgotam, o VBA assume o controlo. Excerto do evento Form_BeforeUpdate que blinda a RN1:

VBA · BeforeUpdate
' Validação da RN1 - Obrigatório presença para Fecho de Acordo
Private Sub Form_BeforeUpdate(Cancel As Integer)
  If Me.Estado_Atual = "Acordo" Then
    If DCount("*", "VISITA", "ID_Lead = " & Me.ID_Lead & _
       " AND Presenca_Confirmada = True") = 0 Then
      MsgBox "Operação bloqueada. Não pode alterar a Lead para 'Acordo' " & _
             "sem pelo menos uma Visita com presença confirmada.", vbCritical
      Me.Undo
      Cancel = True
    End If
  End If
End Sub

— 01
O paradigma analógico

O caos dos dados dispersos.

O atual paradigma do setor imobiliário caracteriza-se por elevada competitividade e ciclos de negociação dependentes de múltiplas variáveis. A gestão informal — cadernos, folhas de cálculo isoladas, programas não interligados — revela-se insuficiente e até destrutiva.

As-Is · Cenário atual

Gestão dispersa

  • Follow-ups esquecidos · perda de leads frias
  • Conflitos de agendamento e sobreposição de visitas
  • Impossibilidade de cruzar perfis de cliente com carteira
  • Dificuldade na gestão temporal de transações simultâneas
  • Métricas inexistentes ou calculadas manualmente
To-Be · Visão SIG KIKO

Inteligência ativa

  • Notificações automáticas e radar de leads estagnadas
  • Validação temporal nativa (datas no passado bloqueadas)
  • Cruzamento relacional Cliente ↔ Lead ↔ Imóvel ↔ Visita
  • Pipeline formalizado em 6 estados To-Be
  • KPIs em tempo real, sem esforço analítico manual

— 05
Os quatro pilares · RN1 a RN4

Integridade não é configuração. É arquitetura.

As quatro regras de negócio formalizadas no projeto transformam o SIG KIKO numa plataforma ativa de validação — não um simples repositório.

RN1

Acordos exigem visita

Matematicamente impossível assinar um acordo sem visita registada com presença efetiva.

RN2

Imóveis com docs

Recusa a inserção de novas angariações sem documentação técnica mínima anexada.

RN3

Qualificação obrigatória

Lead só é ativada após preenchimento estrito de orçamento máximo e zona de preferência.

RN4

Cálculos em runtime

Proibição de armazenamento físico de variáveis dependentes (ex: preço/m²). Cálculos puramente em runtime.

— 06
Modelo conceptual e lógico

Da realidade ao schema relacional.

O Modelo Conceptual identificou quatro entidades basilares com cardinalidades 1:N. A transição para o Modelo Lógico assegurou conformidade com a 3ª Forma Normal.

Cardinalidades MCD

  • Cliente (1) possui Leads (N)
  • Proprietário (1) detém Imóveis (N)
  • Imóvel (1) é alvo de Visitas (N)

Normalização aplicada

  • 1FN · atributos atómicos (um telefone por campo)
  • 2FN · dependência integral da chave primária
  • 3FN · dependências transitivas eliminadas (ex: Preço/m²)

— 09
Desafios encontrados

Os obstáculos do desenvolvimento.

O sucesso de qualquer Sistema de Informação Organizacional não reside unicamente no software escolhido, mas na perfeita aderência entre as ferramentas digitais e o modelo de negócio.

Normalização de moradas

Reconciliação de freguesias urbanas e zonas comerciais alargadas. Resolvido com tabelas de domínio estritas — impossível inserir "LX" e "Lisboa" em simultâneo.

Lógica VBA avançada

Onde as Validation Rules nativas se esgotam, o VBA assume — exigindo desenvolvimento cuidado dos eventos BeforeUpdate e do tratamento de erros.

Equilíbrio simplicidade × robustez

Interface simples para o consultor, mantendo back-office com integridade rigorosa alinhada com princípios da ISO 27001.

Limitação do paradigma desktop

MS Access é formidável como SGBDR pedagógico, mas inviabiliza trabalho colaborativo assíncrono e mobilidade no terreno — daí a evolução proposta.

Evolução proposta

O próximo ciclo: SaaS Cloud-Native.

As portas para o desenvolvimento de um produto SaaS moderno estão abertas. O motor relacional e a modelação operativa estão consolidados. Falta agora envelopá-los numa camada UX premium e fluida.

SGBD Cloud Relacional

Back-End APIs

Node.js / Python absorvendo a lógica VBA. State Machine (XState) para gerir o ciclo da Lead. Validações via Zod/Joi.

Front-End mobile-first

React/Next.js com TailwindCSS. Foco absoluto em mobile-first para acesso em tempo real no terreno.

Integrações futuras

Simuladores de crédito habitação (Banco de Portugal). Módulos georreferenciados de preços por zona. AVMs.

— 05 · Quatro regras que protegem o negócio

O sistema diz não quando algo não bate certo.

RN1
Não há acordo sem visita.
Para fechar um negócio, tem de existir pelo menos uma visita com presença confirmada.
RN2
Imóveis só entram com documentos.
Nenhum imóvel é angariado sem os documentos mínimos anexados.
RN3
Lead qualificada exige zona e orçamento.
Sem estes dois dados, a lead não avança. Evita-se perder tempo em contactos sem critério.
RN4
Preço por m² é sempre calculado.
Nunca é guardado fixo na base de dados. Aparece sempre atualizado, evitando enganos.
